# PR レビュー（Mekajiki スタイル）

組み込みの `/review`（浅い汎用レビュー）や `/code-review`（diff の機械的バグ/整理検出）と違い、
このskillは **設計・スコープ・規約** に踏み込んだ人手レビューを、Mekajiki の型で書く。
機械的なバグ拾い・クリーンアップ検出が欲しい場面は `/code-review` に委譲してよい。

## ワークフロー

0. **セッションタイトルに PR 番号を含める**：対象が確定したら、レビューの最初の出力で `{org}/{repo}#{pr_number}`（例: `mekajiki/dotfiles#123`）を明示し、これがセッションタイトルに反映されるようにする。タイトルを直接設定するツールは無いため、冒頭で必ずこの識別子を出すことで誘導する。
1. **対象を取得**：PR番号 or URL から
   - `gh pr view <n> --json title,body,author,baseRefName,headRefName,state,additions,deletions,files,commits`
   - `gh pr diff <n>`
   - cassette / lockfile / 自動生成物・大量データは精読対象から外すが、**サイズ感は把握**して必要なら指摘する（ノイズ混入もレビュー対象）。
2. **diff だけで判断しない**：呼び出し元・呼び出し先・モデル定義を Read し、引数の意味（例: `total_price` は gross か covered か）やドメイン挙動を確認してから書く。
3. **規約と突合**：グローバル `~/.claude/CLAUDE.md` と対象リポの `CLAUDE.md` を読み、違反を具体的に引用する（例:「CLAUDE.md (Controller設計) に『…』とある」）。
4. **過去の自分のレビューに粒度を合わせる**（任意）：`gh api repos/<owner>/<repo>/pulls/<n>/comments` 等で近いPRの自分のコメントを見て、トーン・タグ運用を揃える。
5. **機械的バグ/整理は委譲可**：必要なら `/code-review` を回し、その結果を取り込みつつ、このskillでは下記「Mekajiki らしさ」に集中する。
6. **分類して書く**（次節）。
7. **投稿は外向き操作**：勝手に投稿しない。レビュー本文を提示し、**投稿してよいか確認してから** `gh pr review` / inline コメントを出す。取捨選択の希望も聞く。

## 出力フォーマット

- **サマリを先頭に**：全体評価（良い点も率直に）→ 何が核心か → 優先度の地図（例:「①②が核心、③以降は nits」）。読み手が優先順位を一目で掴めるようにする。
- 各指摘に **タグ + 短い見出し**：
  - `[要対応]` … 規約違反・バグ・確実に直すべき問題。
  - `[要相談]` … 設計判断・スコープ・トレードオフ。断定せず選択肢（a)/(b)）を示して相談する。
  - `[nits]` … 細かい整理・ネーミング・統一。
- **文章は通常の句読点「、。」**（全角「，．」はコミット/PRメッセージ用の別ルール。レビュー本文には適用しない）。識別子・コマンド名はバッククォートで囲む。
- 過不足ない簡潔さ。冗長な定型（## Test plan やチェックボックス）は使わない。

## Mekajiki らしさ（指摘の中身）

- **root-cause / ゼロベース**：その行を直すだけでなく「責務を分ければこのコード自体が消える」のように、症状でなく原因を指す。一つのモデル/フォーム/controller に責務を集約しすぎていないか（肥大化の兆候）を見る。
- **実行時 altitude も見る（静的な設計の綺麗さだけで通さない）**：変更がプロセス/ホスト境界を跨いでデータや処理を受け渡すなら（リクエスト↔バックグラウンドジョブ、プロセス↔プロセス、サービス↔サービス）、各ステップが依存するリソースがその実行場所から到達可能かを追う。プロセスローカルな状態（ローカルファイル、メモリ、プロセス内キャッシュ）は共有基盤で裏打ちされない限り境界を越えない。あわせて重い/外部I/O がレイテンシ制約のある経路（リクエストサイクル、トランザクション、ロック保持区間）に居ないかを確認する。
- **スコープを問う**：機能と無関係な infra/付随変更が同梱されていたら、別PRへの切り出し、または最低限PR本文での説明を求める。挙動変化（例: TLS 設定の削除）が理由不明なら意図を確認する。
- **規約・既存パターンに寄せる**：「コントローラーで param を加工して弾かない」「ビジネスロジックはモデルへ」等、CLAUDE.md と既存の正しいパターン（例: 既存の validation）を引いて誘導する。
- **テスト設計を見る**：private method を `send` で叩いていないか、不要な VCR カセット等のノイズを生んでいないか、副作用は `expect{}.to change{}` か、観点が網羅されているか。純粋ロジックは HTTP/VCR 抜きでテストできる単位に切り出す提案をする。
- **ネーミングを実態に合わせる**：メソッド/変数名が挙動と食い違っていたら指摘（`is_` prefix を避け述語は `?` 等）。
- **ドメイン不明点は質問で出す**：勝手に正誤を断定せず、「全額0円なので※が落ちて問題ないか確認したい」のように問いとして残す。
- **リリース/ロールアウトまでが実装者の責任か見る**：feature flag や外部連携を伴う変更で、リリース時に必要な作業をリリース担当へ委譲して終わりにしていないか。実装者がロールアウト完遂まで握り、PR に残タスクが書かれているかを確認する。リポ固有のリリース運用は対象リポの CLAUDE.md に従う。

## リポ固有の規約

issue 連携の書式・リリース運用・命名規約など、リポごとの固有ルールは対象リポの `CLAUDE.md` を読んで従う（このskillには特定プロジェクト固有の情報を書かない）。
